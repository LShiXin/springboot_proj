# 定时任务调度服务说明

## 概述

本核心服务实现了定时任务的自动调度功能，每5分钟扫描一次所有定时任务，加载符合条件的任务，更新cron表达式和下次执行时间，并创建执行队列。

## 核心功能

### 1. 定时扫描（5分钟一次）
- 扫描所有 `enabled=true` 的定时任务
- 加载任务开始时间在当前执行周期之内且未到结束时间的任务
- 根据执行间隔更新cron表达式
- 更新下次执行时间
- 创建执行队列

### 2. 数据库更新流程
1. **扫描任务**：查询所有活跃的定时任务配置
2. **更新内存对象**：更新每个任务的cron表达式和下次执行时间
3. **创建执行队列**：按下次执行时间排序，创建执行队列
4. **更新数据库**：执行队列创建完毕后，将更新后的cron_expression和next_fire_time保存到数据库

### 3. 支持的调度类型
- **CRON**：使用标准的cron表达式
- **FIXED_RATE**：固定频率执行（从上一次执行开始计算）
- **FIXED_DELAY**：固定延迟执行（从当前时间开始计算）
- **ONCE**：一次性任务

## API接口

### 启动调度服务
```
POST /api/scheduling/start
```
启动定时任务调度服务，服务启动后会立即执行一次扫描。

### 停止调度服务
```
POST /api/scheduling/stop
```
停止定时任务调度服务。

### 手动触发扫描
```
POST /api/scheduling/scan
```
手动触发一次定时任务扫描。

### 检查服务状态
```
POST /api/scheduling/status
```
检查定时任务调度服务的运行状态。

## 配置说明

### 自动启动
服务配置了自动启动，应用启动时会自动启动定时任务调度服务。相关配置在 `SchedulingAutoStartConfig.java` 中。

### 扫描间隔
默认扫描间隔为5分钟，可在 `TaskSchedulingServiceImpl.java` 中修改 `SCAN_INTERVAL_MINUTES` 常量。

## 数据库更新机制

### 更新时机
数据库更新发生在执行队列创建之后，确保：
1. 所有任务的cron表达式和下次执行时间都已计算完成
2. 执行队列已按执行时间排序
3. 只有发生变化的字段才会更新到数据库

### 更新日志
服务会记录详细的更新日志，包括：
- 哪些任务需要更新
- cron表达式和下次执行时间的变化
- 数据库更新结果

## 执行队列

### 队列创建
1. 按下次执行时间升序排序
2. 过滤掉没有下次执行时间的任务
3. 记录每个任务的执行时间和cron表达式

### 扩展点
当前实现仅记录执行队列，实际项目中可以：
1. 将任务放入Redis队列
2. 发送到消息队列（如RabbitMQ、Kafka）
3. 调用任务执行服务

## 使用示例

### 1. 启动服务
应用启动后，调度服务会自动启动，开始每5分钟扫描一次定时任务。

### 2. 查看日志
查看应用日志，了解调度服务的运行情况：
```
开始执行定时任务扫描...
找到 X 个活跃的定时任务配置
任务 1 需要更新数据库: cron_expression=null -> 0 */5 * * * *, next_fire_time=null -> 2024-01-01T10:05:00
开始创建执行队列，共 X 个任务
任务 1 加入执行队列，计划执行时间: 2024-01-01T10:05:00, cron表达式: 0 */5 * * * *
开始更新数据库，共 X 个任务需要更新
数据库更新完成
定时任务扫描完成，处理了 X 个任务，更新了 Y 个任务到数据库
```

### 3. 通过API管理
可以通过REST API手动管理调度服务：
```bash
# 启动服务
curl -X POST http://localhost:8080/api/scheduling/start

# 停止服务
curl -X POST http://localhost:8080/api/scheduling/stop

# 手动触发扫描
curl -X POST http://localhost:8080/api/scheduling/scan
```

## 注意事项

1. **事务管理**：服务使用 `@Transactional` 注解，确保数据库操作的事务性
2. **并发安全**：使用 `volatile` 关键字保证服务运行状态的可见性
3. **错误处理**：捕获所有异常，避免调度服务因单个任务失败而停止
4. **日志记录**：详细的日志记录便于问题排查和监控
5. **性能考虑**：批量更新数据库，减少数据库连接开销
